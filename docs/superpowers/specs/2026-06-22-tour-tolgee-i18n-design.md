# Tolgee i18n for `apps/tour` — Design

**Date:** 2026-06-22
**Status:** Approved for planning
**Scope:** Internationalize the `apps/tour` admin backoffice UI with Tolgee — full infra plus conversion of every existing static label. Business content (products, etc.) stays owned by `@czo/translation` and is out of scope.

## Goal

Externalize every static UI string in `apps/tour` to Tolgee, render them through the Tolgee React SDK with SSR-correct loading, let an admin switch the UI language (EN/FR), and enable Tolgee's in-context editor in development. Production serves frozen JSON bundled at build time; development talks to the Tolgee API for live edits.

## Key decisions (settled during brainstorming)

| Decision | Choice |
|---|---|
| What is localized | The tour app's **static UI strings** only. Entity content stays on `@czo/translation`. |
| In-context editing | Yes, **dev-only** (Tolgee `DevTools` + API key). Tree-shaken out of prod. |
| Production delivery | **Static JSON bundled** via `@tolgee/cli` pull. No runtime network dependency, no API key shipped. |
| Language selection | **Cookie + switcher now**, account-preference later (code structured to swap the source without rework). |
| Default UI language | **English (`en`)** — aligns with the Tolgee base language. |
| Fallback language | `en`. |
| Available languages | `en`, `fr-FR` (exact Tolgee tags). |
| Switcher placement | **nav-user dropdown** (next to Sign out). Login page is not switchable (defaults to `en`). |
| Key authoring | Pushed to Tolgee **via the Tolgee MCP** during implementation: create keys (EN base), `machine_translate` EN→FR, then `tolgee pull` generates the committed JSON. FR reviewed afterward. |
| SSR load shape | **Direct JSON imports** as `staticData` (no lazy import, no `await tolgee.run()`). SSR and hydration are strictly consistent, no loading flash. |

## Architecture

The Tolgee instance is created once per app boot and provided through `TolgeeProvider` at the router root. It is configured with:

- `availableLanguages: ['en', 'fr-FR']`, `defaultLanguage: 'en'`, `fallbackLanguage: 'en'`. The initial `language` passed at construction is the cookie-resolved locale (default `en`), so SSR renders the right language before any client interaction.
- `staticData: { en, 'fr-FR' }` — both JSON files imported **directly** (eager), so the active language's records are present synchronously during server render. Bundle cost is negligible (two languages, admin-sized string set).
- Plugins: `ReactPlugin()` + `FormatSimple()` always; `DevTools()` added **only** when `import.meta.env.DEV`, together with `apiUrl`/`apiKey` from `VITE_TOLGEE_API_URL` / `VITE_TOLGEE_API_KEY`. With no API key in production, DevTools and the in-context machinery are tree-shaken out.

The active locale flows from a cookie (`czo_locale`), read server-side at the root loader so the first SSR render already uses the right language; `<html lang>` reflects it. The switcher writes the cookie through a server function and re-runs the loaders.

### Data flow

1. Request hits tour → root loader calls `getLocale()` (server fn) → reads `czo_locale` cookie → returns `{ locale }` (default `en`, fallback `en`).
2. `router.tsx` `Wrap` builds the Tolgee instance with the resolved `language` and `staticData`, wraps the tree in `<TolgeeProvider>`.
3. Components call `useTranslate()` → `t('key')`; values come from the eager `staticData` for the active locale.
4. Admin picks a language in nav-user → `tolgee.changeLanguage(tag)` (instant client switch) + `setLocale(tag)` server fn (persists cookie) + `router.invalidate()` (re-sync SSR-derived state, e.g. `<html lang>`).

### In dev vs prod

- **Dev:** `staticData` still loads the committed JSON, but `DevTools()` + API key enable Alt+click in-context editing against the live Tolgee project. New/edited strings show immediately; `i18n:pull` re-freezes them into JSON.
- **Prod:** only `staticData` is active. No API key, no network call, no DevTools.

## File structure

**New files (all under `apps/tour/`):**

- `src/i18n/tolgee.ts` — `createTolgee(language: string)` factory returning the configured Tolgee instance. Holds the plugin/env/staticData wiring described above. One responsibility: instance construction.
- `src/i18n/en.json`, `src/i18n/fr-FR.json` — frozen translations, **generated by `tolgee pull`**, committed. Not hand-edited.
- `src/i18n/locale.server.ts` — `getLocale()` and `setLocale(tag)` server functions over the `czo_locale` cookie. `getLocale` validates against `['en','fr-FR']` and falls back to `en`. This file is the single seam where account-preference resolution will later be added.
- `src/components/locale-switcher.tsx` — the EN/FR control rendered inside nav-user. Uses `useTolgee()` for the current language and `tolgee.changeLanguage` + `setLocale` + `router.invalidate`.
- `tolgee-config.json` (or `.tolgeerc`) — `@tolgee/cli` config: project id, `apiUrl`, pull format (one JSON per language), output dir `src/i18n/`.

**Modified files:**

- `src/router.tsx` — `Wrap` composes `TolgeeProvider` (outermost, then existing `QueryClientProvider`). The instance is built from the root-loader locale.
- `src/routes/__root.tsx` — root loader returns `{ locale }`; `<html lang={locale}>`.
- `src/components/nav-user.tsx` — render `<LocaleSwitcher />` in the dropdown; `Sign out` label becomes `t('nav.signOut')`.
- `src/routes/login.tsx`, `src/components/app-sidebar.tsx`, `src/routes/_authed.tsx`, `src/routes/_authed/index.tsx`, `src/routes/_authed/products/index.tsx`, `src/routes/_authed/products/$productId.tsx` — every hardcoded string → `t('key')`.
- `apps/tour/package.json` — add `@tolgee/react` (dep), `@tolgee/cli` (devDep), script `i18n:pull`.
- `apps/tour/.env.example` — document `VITE_TOLGEE_API_URL`, `VITE_TOLGEE_API_KEY` (dev/in-context), and `TOLGEE_API_KEY` (CLI pull).

## Key convention

Dot-namespaced keys grouped by UI zone. Base language values are **English**.

| Namespace | Covers | Examples |
|---|---|---|
| `login.*` | Login screen | `login.title`, `login.subtitle`, `login.identifierLabel`, `login.passwordLabel`, `login.remember`, `login.forgot`, `login.submit`, `login.invalidCredentials`, `login.brandTagline`, `login.brandBadge`, `login.copyright`, `login.showPassword`, `login.hidePassword` |
| `nav.*` | Sidebar + header + nav-user | `nav.platform`, `nav.dashboard`, `nav.catalog`, `nav.products`, `nav.categories`, `nav.collections`, `nav.attributes`, `nav.toggleCatalog`, `nav.appName`, `nav.signOut`, `nav.language` |
| `dashboard.*` | Dashboard cards + table | `dashboard.title`, `dashboard.subtitle`, `dashboard.sections.products`, `dashboard.sections.categories`, …, `dashboard.soon`, `dashboard.recentProducts.title`, `dashboard.recentProducts.description`, `dashboard.recentProducts.empty`, `dashboard.col.name`, `dashboard.col.handle` |
| `products.*` | Products list/detail | `products.title`, `products.col.name`, `products.col.handle`, `products.morePaginated` |
| `common.*` | Shared/misc | `common.notFound` (router fallback) |

Authoring source of truth is **English**: I write the English base value for every key. For login (currently French) the English base is authored fresh and the existing French copy becomes the `fr-FR` translation. For dashboard/sidebar/products (currently English) the English text is the base and `fr-FR` is machine-translated, then human-reviewed.

Dynamic values (user name, email, product name/handle) are **not** keys — they are interpolated data, rendered as-is.

## Error handling

- `getLocale()` returns `en` for a missing or unrecognized cookie value — never throws.
- Missing key at runtime: `FormatSimple` + `fallbackLanguage: 'en'` resolves to the English value; a key absent from both renders its key string (acceptable, surfaces gaps loudly in dev).
- CLI pull failure (network/auth) is a build-time/dev concern; the committed JSON is the source for prod builds, so a transient pull failure never breaks a build.

## Testing

- `locale.server.test.ts` — `getLocale` returns `en` by default, `en` for an unknown cookie value, the stored tag for a valid cookie; `setLocale` writes the cookie. Pure/unit (vitest), mirrors the existing `auth.server.test.ts` style.
- `locale-switcher` or a representative converted component — rendered under `<TolgeeProvider>` with eager `staticData`, asserts the French value appears when language is `fr-FR` and English when `en`. Proves the provider wiring and key resolution end-to-end without a network call.
- Existing tour tests stay green (no behavior change to auth/products data flow).

## Out of scope

- Account-level locale preference (a later PR; `locale.server.ts` is the seam).
- A login-page language switcher.
- Localizing `@czo/translation`-owned entity content.
- ICU plural/format support — `FormatSimple` covers `{var}` interpolation; ICU is added only if a future string needs plurals.
